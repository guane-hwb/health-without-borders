# ADR: HL7v2 Storage and Integration Strategy

## 1. Context

The "Health Without Borders" system requires interoperability with legacy health systems (such as SIP - Perinatal Information System) using the international HL7v2 standard. Currently, the project utilizes the Google Cloud Healthcare API as the message store. However, due to policies regarding Vendor Lock-in mitigation and long-term cost optimization, alternative architectural approaches are being evaluated.

---

## 2. Evaluated Options

### Option A: Managed Cloud Service (Google Cloud Healthcare API)

*Current state of the project.* The `hl7V2Stores` component of Google Cloud is used to ingest, validate, and store messages generated by the FastAPI backend.

**Pros:**
* **Zero Maintenance (Serverless):** Google manages the underlying infrastructure, security patches, and high availability.
* **Immediate Compliance:** HIPAA certification out-of-the-box, ensuring traceability and an immutable access audit trail.
* **Modern Ecosystem:** Enables automatic, real-time conversion of legacy HL7v2 messages to the modern FHIR standard.
* **Event-Driven:** Native integration with Google Pub/Sub to instantly notify other systems when a new patient record arrives.

**Cons:**
* **Vendor Lock-in:** High dependency on the Google ecosystem. Migrating data in the future requires export efforts.
* **Operational Costs (Opex):** While cheap initially, billing occurs for every message ingested, stored, and queried over its lifetime.
* **Black Box:** Lacks a native GUI for a medical administrator to read HL7 messages directly from the console without exporting.

---

### Option B: Open Source Integration Engine (NextGen / Mirth Connect)

Deploying a custom container with *Mirth Connect*, the open-source industry standard for healthcare message routing and translation.

**Pros:**
* **Total Data Sovereignty:** Clinical data never leaves the infrastructure controlled by the project (ideal for strict NGO/Government regulations).
* **Free License:** The base software is open-source, with no per-message volume costs.
* **Legacy Protocol Support:** The only option listed capable of connecting directly to older hospitals using the MLLP (TCP/IP) protocol, rather than just REST/HTTP.
* **Graphical Interface (Dashboard):** Allows technicians to visually monitor message flow, spot errors, and retry failed deliveries.

**Cons:**
* **Infrastructure and Maintenance Costs:** Although the software is free, it requires a dedicated server (VPS, EC2, or Cloud Run with persistent volumes) that must be paid for and maintained by the team.
* **Learning Curve:** Requires training the engineering team on specialized Java/Rhino-based software.
* **Self-Managed Security:** The team is fully responsible for configuring encryption at rest, backups, and audit logs.

---

### Option C: In-House Development (PostgreSQL + Python Library)

Eliminating third-party services and external engines. The current FastAPI backend validates the message using libraries like `hl7apy` and saves the raw text string into a dedicated table (`hl7_audit_logs`) within the existing `hwb_db` database.

**Pros:**
* **Simplified Architecture:** No new moving parts. Everything lives within the infrastructure the team already masters (PostgreSQL + Cloud Run).
* **Zero Additional Cost:** Leverages the already provisioned database. No extra API charges or additional servers.
* **Zero Dependency:** Absolute control over the codebase and data.

**Cons:**
* **Not a True Integration Engine:** Serves merely as a historical repository (audit trail) but cannot automatically route, alert, or re-transmit messages to other hospitals.
* **Reinventing the Wheel:** Building custom HL7 to JSON or FHIR converters requires months of in-house development—effort that engines like Google or Mirth have already solved.
* **Limited Scalability:** In the long term, storing millions of raw HL7 text strings can degrade the performance of the main transactional database if not partitioned properly.

---

## 3. Summary Matrix

| Feature | Option A (Google Cloud) | Option B (Mirth Connect) | Option C (In-House Postgres) |
| :--- | :--- | :--- | :--- |
| **Initial Setup Cost** | Low | High (Server config) | Medium (Code/Tables) |
| **Long-Term Cost** | Pay-as-you-go | Fixed (Server) | Included in current DB |
| **Maintenance Level**| None | High | Low |
| **Avoids Lock-in** | No | Yes | Yes |
| **FHIR Conversion** | Automatic | Requires custom plugins| Not available |

## 4. Strategic Recommendation

For the current stage (Milestone 2/3 - Interoperability Pilot), **Option A (Google Cloud)** is the most pragmatic approach to demonstrate value quickly without diverting engineering resources toward server maintenance.

If in future phases (National Scaling) the Ministry of Health explicitly requires local MLLP connections or total autonomy on government servers, the current modular architecture allows replacing the Google API endpoint with an **Option B (Mirth Connect)** deployment without affecting the rest of the FastAPI application.
