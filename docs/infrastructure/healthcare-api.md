# Cloud Healthcare API & HL7v2 Configuration

This document details the configuration of the **Google Cloud Healthcare API**, specifically the HL7v2 Store used to ingest, store, and manage clinical messages from external hospital systems (SIP/EHR) in the **UNICEF Border Health** project.

## 1. Architecture Overview

The interoperability layer consists of a hierarchical structure hosted in Google Cloud:

1.  **Dataset:** A top-level container for all healthcare data (FHIR, HL7v2, DICOM) located in a specific region.
2.  **HL7v2 Store:** A specific data store within the dataset designed to ingest and parse HL7v2 messages.

* **Region:** `us-central1` (Must be co-located with Cloud Run and Cloud SQL for low latency).
* **Standard:** HL7v2 (Pipe-delimited format).

---

## 2. Resource Provisioning

### 2.1. Enable the API
The Cloud Healthcare API must be enabled in the project.

```bash
gcloud services enable healthcare.googleapis.com

```

### 2.2. Create the Dataset

The dataset acts as the administrative boundary for compliance and data residency.

* **ID:** `migrants_dataset`
* **Location:** `us-central1`

```bash
gcloud healthcare datasets create migrants_dataset \
    --location=us-central1

```

### 2.3. Create the HL7v2 Store

The store is configured to accept and parse messages.

* **ID:** `unicef_store`
* **Parser Version:** V3 (Recommended for better compatibility).
* **Reject Duplicates:** Disabled (to allow updates/corrections to existing messages).

```bash
gcloud healthcare hl7v2-stores create unicef_store \
    --dataset=migrants_dataset \
    --location=us-central1 \
    --parser-version=V3

```

---

## 3. Access Control (IAM)

To allow the Backend API (Cloud Run) or the ingestion scripts to write messages to the store, specific IAM roles are required.

### 3.1. Service Account Permissions

The identity running the ingestion process (e.g., `backend-uploader@...`) must have the following roles:

| Role Name | ID | Purpose |
| --- | --- | --- |
| **Healthcare HL7v2 Store Editor** | `roles/healthcare.hl7V2StoreEditor` | Allows creating, ingesting, and deleting HL7v2 messages. |
| **Healthcare Dataset Viewer** | `roles/healthcare.datasetViewer` | Allows listing datasets and stores to verify existence. |

### 3.2. Assigning Roles via CLI

```bash
# Example: Granting access to the Compute Engine default service account (or your specific SA)
gcloud projects add-iam-policy-binding migrants-unicef \
    --member="serviceAccount:YOUR_SERVICE_ACCOUNT_EMAIL" \
    --role="roles/healthcare.hl7V2StoreEditor"

gcloud projects add-iam-policy-binding migrants-unicef \
    --member="serviceAccount:YOUR_SERVICE_ACCOUNT_EMAIL" \
    --role="roles/healthcare.datasetViewer"

```

## 4. Message Ingestion & Verification

### 4.1. Ingestion Endpoint

The standard REST endpoint for creating (ingesting) a message is:

`POST https://healthcare.googleapis.com/v1/projects/migrants-unicef/locations/us-central1/datasets/migrants_dataset/hl7V2Stores/unicef_store/messages`

### 4.2. Base64 Encoding

The HL7 message content must be **Base64 encoded** within the JSON payload.

**Example Payload (`message.json`):**

```json
{
  "message": {
    "data": "TVnDiA... (Base64 encoded MSH segment)..."
  }
}

```

### 4.3. Manual Verification (cURL)

To list ingested messages and verify connectivity from a local terminal authorized via `gcloud`:

```bash
curl -X GET \
    -H "Authorization: Bearer $(gcloud auth print-access-token)" \
    "[https://healthcare.googleapis.com/v1/projects/migrants-unicef/locations/us-central1/datasets/migrants_dataset/hl7V2Stores/unicef_store/messages?view=FULL](https://healthcare.googleapis.com/v1/projects/migrants-unicef/locations/us-central1/datasets/migrants_dataset/hl7V2Stores/unicef_store/messages?view=FULL)"

```

## 5. Operations

* **Monitoring:** Message counts and ingestion errors can be monitored in the [Cloud Healthcare Console](https://console.cloud.google.com/healthcare/browser).
* **Exporting Data:** Raw messages can be exported to **Google Cloud Storage (GCS)** buckets via the console for auditing or bulk analysis.
* **Compliance:** Access logs are automatically generated by Google Cloud Audit Logging if configured in the project settings.
